Hands-on. Een data-steward vraagt een nieuwe rol `dq_inspector`: read-only
toegang tot DQ-overzichten, geen cliënt-PII. Klinkt simpel — maar als je
het verkeerd uitrolt zie je het pas op productie.

## Stap 1 — role-mapping ontwerpen

Open
[uwv_role_mappings.json](https://vscode.dev/github.com/UWV/UDP_Stackable/blob/main/opa-policies-src/data/uwv_role_mappings.json)
in vscode.dev. Voeg toe:

```json
"dq_inspector": {
  "catalogs": ["openmetadata", "gold"],
  "schemas": ["dq", "uc01_wia_funnel", "uc06_lastprognose"],
  "purposes": ["dq_review"],
  "can_see_pii": false,
  "can_see_medical": false,
  "can_see_bankrekening": false,
  "regio_filter": false,
  "break_glass": false,
  "_role_purpose": "Read-only DQ-rapportages voor data-stewards-in-opleiding."
}
```

## Stap 2 — Rego-test

In `opa-policies-src/trino/policy_test.rego` voeg je toe:

```rego
test_dq_inspector_no_pii {
  not allow_select with input as {
    "user": "dq.test", "groups": ["dq_inspector"],
    "resource": "silver.wia.aanvraag.bsn"
  }
}

test_dq_inspector_can_dq {
  allow_select with input as {
    "user": "dq.test", "groups": ["dq_inspector"],
    "resource": "openmetadata.dq.test_results"
  }
}
```

`make opa-test`. Groen voor je gaat verder.

## Stap 3 — PR aanmaken & laten reviewen

Reviewers: data-steward (intentie) + tweede platform-admin (technisch).
Geen self-merge. Bewust: de OPA-policy raakt iedereen — dus 4-eyes.

## Stap 4 — Merge → bundle-reload

Na merge:

```bash
make opa-bundle-reload
# Checkt: alle Trino-coordinators picken nieuwe bundle binnen 30s
```

Verifieer in [Prometheus](/go/prometheus/): `opa_bundle_loaded_timestamp`
moet hoger staan dan vóór de reload, op alle replicas.

## Stap 5 — Keycloak group + test-user

In [Keycloak](/go/keycloak/) → realm `uwv` → Groups → "dq_inspector"
aanmaken. Daarna een test-user `dq.test` met deze group.

> Test-user is geen permanente account. Doel: de rol werkend bewijzen op een
> echte applicatie, niet alleen in de YAML.

## Stap 6 — Smoke-test als test-user

```bash
KEYCLOAK_USER=dq.test make test SUITE=role-smoke ROLE=dq_inspector
```

Dit draait een set queries:

- toegang tot `gold.dq.*` — moet werken;
- toegang tot `silver.wia.aanvraag.bsn` — moet 0 rijen geven;
- toegang tot `sensitive.wajong.*` — moet 0 rijen geven.

Alle drie groen? Verder. Een falend signaal? Revert (zie volgende stap).

## Stap 7 — Communicatie

E-mail data-steward: "Rol uitgerold, smoke-test groen, test-user
`dq.test` actief tot maandag voor jullie eigen validatie. Productie-users
toevoegen als jullie tevreden zijn."

## Wat als bundle-reload faalt?

Niet panicken. Niet in productie debuggen. Revert via een nieuwe commit
(geen force-push); audit-spoor moet kloppen. Onderzoek de fout in een
worktree. Het oude bundle blijft actief, dus je gebruikers merken niets.

## Verificatie

De auto-check valideert dat de role-smoke-test groen is voor `dq_inspector`
en dat de bundle-reload op alle replica's is doorgekomen.
